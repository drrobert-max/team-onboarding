01

What you're deploying

The Training Hub is an internal web app for onboarding and training clinic staff. It's a Progressive Web App — staff open it in a browser and can "install" it to their phone's home screen, so it feels like a native app without an App Store.

Out of the box it gives a new office:

📋 SOPs that stay current

Standard operating procedures pulled straight from your Google Docs — edit the doc, the app updates itself and re-flags staff to re-read.

🗺️ Learning tracks

Role-based training paths (assistant, associate doctor, scan tech…) with modules, progress tracking, and daily focus prompts.

✅ Quizzes & test-outs

Auto-generated quizzes per module (optional AI), plus "test out" checkpoints to prove competence.

🎥 Video library & submissions

A synced library of training videos, plus staff submitting their own technique videos for a preceptor to review and sign off.

One place a new hire logs into on day one — and the same place they prove they're ready.

You own the data.It's your GitHub repo, your database, your Drive — not a rented SaaS seat you lose access to.
Cheaper than one SaaS license.A whole clinic runs for roughly the price of a single seat on most training platforms.
Content lives in Google.Non-technical staff edit SOPs and drop in videos using tools they already know.
Installs on any phone.No app store, no approvals — share a link, staff add it to their home screen.

The key idea

Railway watches your GitHub repo. Every time code lands on the main branch, Railway rebuilds and redeploys automatically. There's no "server" you log into and babysit — you change code or content, and the live site updates on its own.

Why the LLM cost is ~$0

Quiz generation calls the AI once per module, ever — the result is cached in your database. A whole training catalog is pennies. The app is deliberately built to never make looping or per-view AI calls, so this line stays near zero.

* Storage is "required" only if you want video submissions and uploaded media. The library-video feature indexes files that live in Google Drive, so a very lean deployment can defer S3 until you need uploads.

04

Accounts to open first

Register these before you start. The two dashed ones are optional — you can add them later without redeploying.

GitHub Need

Free account at github.com. Holds the code and triggers deploys. You'll copy (fork) the existing repo into your own account.

Railway Need

Sign up at railway.app with your GitHub account. This hosts the app and provides the MySQL database as a one-click plugin. Add a payment method.

Cloudflare (R2) Need*

Free Cloudflare account, then enable R2 storage. Generous free tier. AWS S3 works identically if you prefer it.

Google account Need

A Google account with Drive — ideally a clinic account, not personal. Holds your SOP docs and the training-video folder.

OpenAI Optional

platform.openai.com. Only needed for auto-generated quizzes. Add ~$5 of credit and it'll last a very long time.

Gmail app password Optional

Turn on 2-factor auth on a Gmail account, then generate an "app password." Enables password-reset and notification emails.

05Phase 1 — Get the code

Copy the repository

  1. Sign into GitHub.
  2. Go to the source repo and click Fork (top-right) to create your own copy under your account.
  3. That's it — you now own an independent copy you can deploy and customize.
Why fork instead of share

A fork gives the new office its own code, its own deploy pipeline, and the freedom to change branding and content without touching anyone else's live site.

06Phase 2 — Database & hosting

Create the project on Railway

  1. In Railway, click New Project → Deploy from GitHub repo and pick your fork. Railway reads railway.json and knows how to build it (pnpm build) and start it (pnpm start), with a health check at /healthz.
  2. In the same project, click New → Database → Add MySQL. Railway provisions it and exposes a connection string.
  3. Open the MySQL service → Variables → copy the DATABASE_URL value. You'll paste it into the app in the next phase.
Requirements are already declared

The app targets Node ≥ 20.11 and pnpm 10, both handled automatically by Railway's Nixpacks builder. Nothing to install by hand.

07Phase 3 — Configure & deploy

Set environment variables

Open your app service on Railway → Variables. These three are the minimum to boot:

VariableValueNotes
DATABASE_URLfrom the MySQL pluginReference the MySQL service's variable
JWT_SECRETa long random stringSigns login cookies — keep it secret
RUN_MIGRATIONStrueBuilds the database tables on first boot

Then add the feature variables you want. Any you leave out simply stay dormant — the feature is hidden, nothing breaks:

FeatureVariables
Video / image storageS3_BUCKET, S3_REGION, S3_ACCESS_KEY_ID, S3_SECRET_ACCESS_KEY — plus S3_ENDPOINT & S3_FORCE_PATH_STYLE=true for Cloudflare R2
Quiz generationLLM_API_KEY, LLM_BASE_URL, LLM_MODEL (e.g. gpt-4o-mini)
EmailGMAIL_USER, GMAIL_APP_PASSWORD, OWNER_EMAIL
Weekly content syncENABLE_CRON=true — runs the Drive syncs on a schedule
Video library from DriveGOOGLE_API_KEY (public folder) — or OAuth: GOOGLE_DRIVE_CLIENT_ID/_SECRET/_REFRESH_TOKEN

Deploy

Saving variables triggers a deploy. Watch the build logs — a healthy boot prints the migration steps, then Server running on …. When the health check at /healthz goes green, your site is live at the Railway-provided URL.

Heads up — this is production

Once deployed, every push to main goes live immediately. For anything non-trivial, work on a branch and open a pull request so nothing reaches staff until you've checked it.

08Phase 4 — Your first login

Create the first admin account

There's no public sign-up. A fresh database has no users, so there's a one-time bootstrap to create the very first admin. It permanently disables itself the moment an admin exists.

  1. On Railway, add a variable SETUP_SECRET with any random value, and let it redeploy.
  2. Send a one-time request to the bootstrap endpoint, passing that secret as a header:
# replace the URL, secret, and your details
curl -X POST https://YOUR-APP.up.railway.app/api/setup/first-admin \
  -H "Content-Type: application/json" \
  -H "x-setup-secret: YOUR_SETUP_SECRET" \
  -d '{"email":"you@clinic.com","password":"a-strong-password","name":"Your Name"}'
  1. You'll get back { ok: true }. Log into the site with those credentials — you'll now see the Admin section (Team, Tracks, Activity, Inbox).
  2. From then on, create all other staff from Admin → Team. The bootstrap endpoint will refuse any further use.
After bootstrapping

You can delete the SETUP_SECRET variable — it's done its job and the endpoint is now permanently closed.

09Phase 5 — Your content

Load your SOPs and videos

Content doesn't live in the app — it lives in Google, and the app syncs it. Two separate pipelines:

SOPs — from Google Docs

Each SOP is a Google Doc. The app fetches the plain-text export, so no API key is needed — the doc just has to be shared publicly for reading.

  1. Write each SOP as a Google Doc.
  2. Share it as "Anyone with the link → Viewer."
  3. Copy the document ID from its URL (the long string in /document/d/THIS_PART/edit).

Video library — from a Drive folder

Drop training videos into one Google Drive folder. The sync lists the folder and registers each video, auto-sorting it into a category based on keywords in the filename (e.g. a file named "Sleep — evening routine" lands in a Sleep category).

  • Easiest auth: share the folder "Anyone with link," and set GOOGLE_API_KEY. Free, no OAuth dance.
  • For private folders: set up OAuth refresh-token credentials instead.
  • Name files with clear keywords so auto-categorization works in your favor.
Keeping content fresh

With ENABLE_CRON=true, the app re-syncs weekly on its own. Edit an SOP doc and staff are automatically re-flagged to re-read it. You can also trigger a sync on demand from the admin tools.

10Phase 6 — The honest caveat

What a new office must change in the code

This app was built for one specific clinic, then generalized. It is not yet a turnkey, multi-tenant product — a second office needs to edit a handful of hardcoded values in its fork. This is the real work of rolling out, so plan for it:

What to changeWhere
Your SOP Google Doc IDs & titlesserver/scheduledSopSync.ts
Your training-video Drive folder IDserver/scheduledLibrarySync.ts
Video auto-categories (keywords)server/scheduledLibrarySync.ts
Logo, app name, colorsclient/public/ & manifest.json
Learning tracks & rolesCreated in-app via Admin, once live
Set expectations

None of this is hard, but it does mean a second office needs someone comfortable editing a few code files (or an assistant like Claude Code to do it for them). Budget an afternoon, not five minutes. If you're rolling out to many offices, the real next investment is making these values configurable in the admin panel instead of the code.

11Ongoing

Keeping it running

🚀 Deploys

Automatic on every push to main. Railway rebuilds and swaps in the new version.

🩺 When something breaks

Railway's deploy logs are the first place to look — boot messages show migrations and server start.

💾 Database migrations

Run automatically on boot when RUN_MIGRATIONS=true. Schema changes ship as committed SQL files.

👥 Adding staff

Admins create every account from Admin → Team. There is no public sign-up by design.

12

Is this right for your office?

✓ A great fit if…

You want to own your training system outright, you already run on Google Docs/Drive, you have someone (or Claude) willing to do a one-time technical setup, and you'd rather pay ~$15/mo than per-seat SaaS fees.

✗ Think twice if…

You need a polished no-code admin for everything on day one, you can't touch a config file or spare an afternoon, or you need multi-office tenancy today without any development.

A clinic-grade training platform you own, for about the price of one streaming subscription.

The heavy engineering is already done. Rolling out to a new office is mostly opening accounts, pasting a dozen settings, and pointing it at your own Google Drive. This playbook is the whole map.
